# 원티드 백엔드 프리 온보딩 챌린지 사전 과제

## 1. 동시에 같은 DB Table row 를 업데이트 하는 상황을 방어하기 위해 어떻게 개발하실 건지 설명해주세요
동시에 같은 데이터베이스 테이블의 행(row)을 업데이트하는 상황은 "race condition"이라고 불리는 문제를 일으킬 수 있습니다.

1. **행 레벨 잠금 (Row-Level Locking)**:
   - 대부분의 관계형 데이터베이스 관리 시스템(RDBMS)은 트랜잭션 중에 특정 행에 대한 접근을 제어하기 위해 행 레벨 잠금을 제공합니다.
   - 예를 들어, SQL에서 `SELECT ... FOR UPDATE` 구문을 사용하면 해당 행에 대한 잠금을 얻을 수 있으며, 다른 트랜잭션이 해당 행을 수정하지 못하도록 합니다.
   - 이 방법은 트랜잭션이 완료될 때까지 다른 트랜잭션의 접근을 차단하여 동시 업데이트 문제를 방지합니다.

2. **낙관적 동시성 제어 (Optimistic Concurrency Control, OCC)**:
   - 낙관적 동시성 제어는 데이터를 읽을 때 잠금을 사용하지 않고, 대신 업데이트 시점에 데이터가 변경되었는지를 확인합니다.
   - 데이터에 버전 번호나 타임스탬프를 추가하여 관리할 수 있습니다. 업데이트를 시도하기 전에 버전 번호나 타임스탬프를 확인하여 데이터가 읽힌 이후 변경되지 않았는지 검사합니다.
   - 변경이 감지되면 업데이트를 거부하고, 애플리케이션은 적절한 조치(예: 재시도, 사용자에게 알림 등)를 취할 수 있습니다.

3. **비관적 동시성 제어 (Pessimistic Concurrency Control, PCC)**:
   - 비관적 동시성 제어는 데이터를 읽는 즉시 잠금을 걸어 다른 트랜잭션이 해당 데이터에 접근하지 못하도록 하는 방법입니다.
   - 이 방법은 동시성이 낮은 환경에서 유용하며, 데이터가 자주 충돌할 것으로 예상될 때 적합합니다.

4. **트랜잭션 격리 수준 (Transaction Isolation Levels)**:
   - 데이터베이스는 다양한 트랜잭션 격리 수준을 제공하여 동시성을 제어할 수 있습니다.
   - 격리 수준을 높이면 동시성은 감소하지만, 데이터 일관성은 증가합니다.
   - 예를 들어, `SERIALIZABLE` 격리 수준은 가장 엄격한 수준으로, 트랜잭션이 순차적으로 실행되는 것처럼 보장하지만, 동시성이 크게 감소합니다.

5. **응용 프로그램 로직**:
   - 애플리케이션 측에서도 동시 업데이트를 방지하기 위한 로직을 구현할 수 있습니다.
   - 예를 들어, 업데이트를 수행하기 전에 애플리케이션 레벨에서 잠금을 구현하거나, 업데이트가 발생할 때마다 중앙 집중식 서비스를 통해 동기화를 수행할 수 있습니다.

6. **리트라이 메커니즘**:
   - 업데이트 중 충돌이 감지되면, 애플리케이션은 자동으로 트랜잭션을 재시도할 수 있는 로직을 포함할 수 있습니다.
   - 이 방법은 충돌이 자주 발생하지 않는 시스템에서 효과적일 수 있습니다.

## 2. TCP 와 UDP 의 차이를 작성해주세요
TCP(Transmission Control Protocol)와 UDP(User Datagram Protocol)는 인터넷 프로토콜 스위트의 핵심 요소로, 데이터를 네트워크를 통해 전송하는 방법을 정의합니다. 이 두 프로토콜은 다음과 같은 주요 차이점을 가지고 있습니다:

1. **연결 지향성 vs 비연결성**:
   - **TCP**는 연결 지향적 프로토콜입니다. 데이터 전송을 시작하기 전에 통신을 위한 연결 설정 과정을 거치며, 이 과정에서 '3-way handshake'라는 과정을 통해 연결을 확립합니다.
   - **UDP**는 비연결성 프로토콜입니다. 연결 설정 없이 데이터를 바로 보낼 수 있으며, 이는 데이터를 보내거나 받는 측에서 연결을 유지할 필요가 없음을 의미합니다.

2. **데이터 신뢰성**:
   - **TCP**는 높은 신뢰성을 제공합니다. 데이터가 손실되거나 순서가 바뀌었을 때 이를 감지하고 재전송을 요청하여 데이터가 정확하게 도착하도록 보장합니다.
   - **UDP**는 신뢰성이 낮습니다. 데이터 패킷이 손실되거나 순서가 뒤바뀌어도 재전송을 하지 않습니다. 이는 일부 데이터가 도착하지 않거나 순서대로 도착하지 않을 수 있음을 의미합니다.

3. **데이터 순서 보장**:
   - **TCP**는 데이터가 전송된 순서대로 도착하도록 보장합니다. 패킷이 잘못된 순서로 도착하면 TCP는 이를 재정렬합니다.
   - **UDP**는 데이터 순서를 보장하지 않습니다. 데이터는 어떤 순서로든 도착할 수 있으며, 순서 조정은 애플리케이션 레벨에서 처리해야 합니다.

4. **속도와 효율성**:
   - **TCP**는 오류 검사와 순서 보장 등의 추가 기능으로 인해 UDP보다 느릴 수 있습니다. 이러한 기능들은 네트워크 오버헤드를 증가시키고 처리 시간을 늘립니다.
   - **UDP**는 최소한의 오버헤드만 가지고 있어 TCP보다 빠르며, 실시간 애플리케이션(예: 스트리밍, 게임)에 적합합니다.

5. **흐름 제어 및 혼잡 제어**:
   - **TCP**는 흐름 제어와 혼잡 제어 메커니즘을 가지고 있어 네트워크의 상태에 따라 데이터 전송 속도를 조절합니다.
   - **UDP**는 이러한 메커니즘이 없어 네트워크의 혼잡 상태에 관계없이 일정한 속도로 데이터를 전송합니다.

6. **사용 사례**:
   - **TCP**는 이메일, 웹 브라우징, 파일 전송과 같이 정확한 데이터 전송이 중요한 애플리케이션에 주로 사용됩니다.
   - **UDP**는 실시간 비디오나 오디오 스트리밍, VoIP, 온라인 게임과 같이 일부 패킷 손실이 허용되고 빠른 전송이 중요한 애플리케이션에 주로 사용됩니다.

## 3. 웹 브라우저에 네이버 를 검색하고 화면에 네이버 화면이 출력이 될 때 까지 내부적으로 어떤 동작들이 수행이 되는지 설명해주세요

1. **사용자 입력**:
   - 사용자가 웹 브라우저의 주소 표시줄이나 검색창에 "네이버"를 입력합니다.

2. **도메인 이름 해석 (DNS Lookup)**:
   - 브라우저는 "네이버"라는 텍스트가 URL인지 검색어인지를 판단합니다. 검색어로 판단되면 기본 검색 엔진(예: Google, Bing 등)을 사용하여 검색을 수행합니다.
   - 만약 사용자가 "naver.com"과 같이 정확한 URL을 입력했다면, 브라우저는 해당 도메인 이름에 대한 IP 주소를 찾기 위해 DNS 서버에 쿼리를 보냅니다.

3. **TCP 연결**:
   - IP 주소가 확인되면, 브라우저는 해당 서버와의 TCP 연결을 설정하기 위해 SYN 패킷을 보냅니다. 이는 일반적으로 80번(HTTP) 또는 443번(HTTPS) 포트를 사용합니다.

4. **HTTP/HTTPS 요청**:
   - TCP 연결이 설정되면, 브라우저는 HTTP 또는 HTTPS 프로토콜을 사용하여 웹 서버에 웹 페이지 데이터를 요청합니다.

5. **서버 처리**:
   - 웹 서버는 브라우저의 요청을 받고, 요청된 URL에 대한 데이터를 처리합니다. 이는 정적 파일을 제공하거나, 필요에 따라 데이터베이스 조회 등의 동적 처리를 포함할 수 있습니다.

6. **HTTP/HTTPS 응답**:
   - 처리가 완료되면, 웹 서버는 HTML, CSS, JavaScript 등의 웹 페이지 데이터를 포함한 HTTP 응답을 브라우저로 보냅니다.

7. **데이터 수신 및 렌더링**:
   - 브라우저는 서버로부터 데이터를 수신하고, HTML을 파싱하여 DOM(Document Object Model) 트리를 구성합니다.
   - CSS는 파싱되어 스타일 규칙이 적용되고, JavaScript는 실행됩니다.
   - 이 모든 요소들이 결합되어 브라우저에 웹 페이지가 렌더링됩니다.

8. **외부 리소스 로딩**:
   - 웹 페이지에는 이미지, 스크립트, CSS 파일 등의 외부 리소스가 포함될 수 있습니다. 이러한 리소스들은 별도의 HTTP 요청을 통해 로드되며, 각각의 리소스가 로드될 때마다 브라우저는 해당 리소스를 렌더링된 페이지에 적절한 위치에 표시합니다.

9. **사용자 상호작용**:
   - 페이지가 완전히 로드되면, 사용자는 링크 클릭, 폼 제출 등의 상호작용을 할 수 있습니다. 이러한 상호작용은 추가적인 HTTP 요청을 발생시킬 수 있으며, 이 과정은 위의 단계를 반복합니다.

## 4. golang에서 설계적 결함 한 가지를 작성해주세요
Go 언어(또는 Golang)는 간결함, 높은 성능, 그리고 동시성을 쉽게 다룰 수 있는 기능 등으로 많은 인기를 얻고 있습니다. 그러나 모든 프로그래밍 언어와 마찬가지로, Go도 설계상의 결함이나 논란이 되는 부분이 있습니다. 한 가지 예로 **타입 시스템과 제네릭(generics)의 부재**가 있습니다.

Go 1.x 버전에서는 제네릭이 지원되지 않았습니다. 이는 Go의 타입 시스템에서 자주 지적되는 한계 중 하나였습니다. 제네릭이 없기 때문에, 개발자들은 타입 안전성을 잃지 않으면서 코드를 재사용하기 위해 인터페이스와 리플렉션을 사용해야 했습니다. 이는 때때로 성능 저하를 초래하고, 코드의 복잡성을 증가시키며, 타입 안전성을 완전히 보장하지 못하는 경우가 있었습니다.

예를 들어, 여러 타입에 대해 동작하는 함수를 작성하려면, 인터페이스를 사용하고 타입 단언(type assertions) 또는 리플렉션(reflection)을 사용해야 했습니다. 이는 코드를 더 복잡하게 만들고, 런타임 오류의 가능성을 증가시키며, 성능에 부정적인 영향을 미칠 수 있습니다.

```go
func PrintAll(vals []interface{}) {
    for _, val := range vals {
        fmt.Println(val)
    }
}

// 사용 예:
PrintAll([]interface{}{1, "hello", true, 2.5})
```

위 예제에서 `PrintAll` 함수는 `interface{}` 슬라이스를 매개변수로 받아, 어떤 타입의 슬라이스든 처리할 수 있습니다. 그러나 이는 타입 안전성을 잃게 만들고, 호출하는 측에서 타입 단언을 필요로 합니다.

이러한 문제를 해결하기 위해 Go 팀은 Go 1.18 버전에서 제네릭을 도입하기로 결정했습니다. 이는 `type parameters`를 사용하여 구현되며, 이를 통해 개발자들은 타입 안전성을 유지하면서 코드를 재사용할 수 있게 되었습니다.

```go
func PrintAll[T any](vals []T) {
    for _, val := range vals {
        fmt.Println(val)
    }
}

// 사용 예:
PrintAll([]string{"hello", "world"})
PrintAll([]int{1, 2, 3})
```

## 5. golang에서 자료구조와 관련 된 클래스가 내부적으로 어떻게 동작하는지 한 가지 사례를 정하여 작성해주세요
Go 언어는 클래스 기반의 객체지향 프로그래밍 언어가 아니며, "클래스"라는 개념을 직접적으로 사용하지 않습니다. 대신, Go는 구조체(structs)와 인터페이스(interfaces)를 사용하여 객체지향 프로그래밍의 일부 개념을 구현합니다. 자료구조를 구현할 때도 이러한 구조체와 인터페이스를 사용합니다.

예를 들어, Go에서 링크드 리스트(Linked List)를 구현하는 방법을 살펴보겠습니다. 링크드 리스트는 각 노드가 데이터와 다음 노드에 대한 참조(또는 포인터)를 가지고 있는 선형적인 자료구조입니다.

다음은 Go에서 단일 연결 리스트(Singly Linked List)의 간단한 구현 예시입니다:

```go
package main

import "fmt"

// Node 구조체는 리스트의 노드를 정의합니다.
type Node struct {
    Value int
    Next  *Node
}

// LinkedList 구조체는 리스트의 머리(head)를 가지고 있습니다.
type LinkedList struct {
    Head *Node
}

// Append 메서드는 리스트의 끝에 새로운 노드를 추가합니다.
func (l *LinkedList) Append(value int) {
    newNode := &Node{Value: value}
    if l.Head == nil {
        l.Head = newNode
        return
    }
    current := l.Head
    for current.Next != nil {
        current = current.Next
    }
    current.Next = newNode
}

// Display 메서드는 리스트의 모든 값을 출력합니다.
func (l *LinkedList) Display() {
    current := l.Head
    for current != nil {
        fmt.Printf("%d -> ", current.Value)
        current = current.Next
    }
    fmt.Println("nil")
}

func main() {
    list := LinkedList{}
    list.Append(1)
    list.Append(2)
    list.Append(3)

    list.Display() // 출력: 1 -> 2 -> 3 -> nil
}
```

위 코드에서 `Node` 구조체는 링크드 리스트의 각 노드를 나타냅니다. `LinkedList` 구조체는 리스트의 시작점인 `Head` 노드를 가지고 있습니다. `Append` 메서드는 새로운 값을 리스트의 끝에 추가하고, `Display` 메서드는 리스트의 모든 값을 순회하며 출력합니다.

이 예시는 Go에서 자료구조를 어떻게 구현할 수 있는지 보여줍니다. Go의 구조체와 메서드를 사용하여 클래스와 유사한 패턴을 만들 수 있지만, 상속이나 클래스 계층 구조는 지원하지 않습니다. 대신, 인터페이스를 통해 다형성을 제공하고, 구조체 임베딩을 통해 상속과 유사한 기능을 모방할 수 있습니다.
